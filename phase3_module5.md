## 1. Module Overview

### Why AI summaries matter
Chat answers are ephemeral — they answer one question and scroll away. Research needs **durable,
structured artifacts**: an executive overview of a paper, a study-revision digest of a chapter, a
unified brief across several documents. The AI Summary Engine produces exactly that — a
**persistent, sectioned, cited summary** that lives in the workspace, can be reopened, renamed,
regenerated, exported, and (later) converted into notes/flashcards.

### How they improve productivity
- **Read less, understand more** — a grounded overview of a 300-page book in seconds.
- **Cross-document synthesis** — one brief that merges several sources while preserving each
  source's citations.
- **Reusable knowledge** — summaries are assets, not chat scrollback; searchable, sortable,
  exportable.
- **Trust** — every section is grounded in retrieved evidence and carries clickable citations into
  the PDF viewer.

### Chat responses vs persistent summaries
| Chat response (Module 4) | Persistent summary (this module) |
|---|---|
| One question → one answer | A structured, multi-section document |
| Lives in a conversation thread | A first-class, listable/searchable asset |
| Synchronous, single LLM turn | **Asynchronous**, multi-section, hierarchical generation |
| Scope = whatever the message implies | Explicit scope: **document / multi-doc / workspace** |
| Transient | Reopen, rename, regenerate, duplicate, export |

Crucially, both reuse the **same** retrieval + context pipeline — a summary is many grounded
retrieval-and-summarize passes, never a raw document dump.

## 2. Previous Architecture

LexiMind could *answer* and *converse* over documents (Modules 1–4) but could not produce a
**standalone, persistent summary**. The `Workspace.summary_count` counter existed (Module 1) but
nothing wrote summaries.

```
Ask / Chat  →  /query or /conversations  →  answer (transient)  →  scrolls away
```

| Concern | Before Module 5 |
|---|---|
| Summary entity | none — no `summaries`/`sections`/`citations` tables |
| Whole-document understanding | only via ad-hoc questions; nothing structured |
| Multi-document synthesis | none |
| Background/async generation | none — everything was request-blocking |
| Persistence / reuse | none — answers weren't saved as assets |

### Limitations
1. **No durable knowledge artifacts** — you couldn't "summarize this book" and keep the result.
2. **No structure** — answers were single blobs, not sectioned documents.
3. **No async** — a long, multi-section generation would block the request/UI.
4. **No multi-doc synthesis** — no way to brief across a selection or a whole workspace.

> **The latent seam:** the retrieval + context pipeline and `structured_citations` already produced
> grounded, cited text per query. Module 5 *orchestrates many such passes* into a persistent,
> sectioned artifact — it does not add a new AI path.

## 3. New Architecture

```
Document(s) / selection / workspace
        ↓  (scope → vector document ids)
Section plan   (per summary type; for detailed/chapterwise derived from the doc's own headings)
        ↓  for each section:
Retrieval Engine   (Phase 1 — workspace-scoped, archived excluded, doc-scoped)
        ↓
Context Engineering (Phase 2 — dedup→rank→budget→compress→assemble, citations preserved)
        ↓
Structured summary prompt  →  LLM (Ollama)
        ↓
structured_citations(evidence)  →  SummarySection + SummaryCitations (persisted)
        ↓  (detailed/chapterwise) intermediate sections → final synthesis section
Persistent Summary  (status: queued → processing → completed)   ← generated in the BACKGROUND
```

### Async by construction
Generation is **asynchronous**: the API creates a `queued` `Summary` and hands its id to a
background runner (a thread pool); the client polls `GET .../status`. Large summaries never block
the UI; cancellation and retry are status transitions the worker observes.

### One pipeline, injected — never duplicated
The summaries module contains **no retrieval/context code**. An injected `SummaryEngine`
orchestrates the existing singletons (`RetrievalPipeline`, `ContextBuilderService`,
`answer_service`) once per section. The production engine imports those heavy singletons lazily, so
`app.summaries.*` imports faiss-free and the entire generation pipeline is unit-testable against a
fast fake engine + an inline runner.

## 4. Database Design

Three **new tables** (created by `create_all`; no migration).

### Table: `summaries`
| Column | Type | Notes |
|---|---|---|
| `id` | `String(40)` PK | `sum_<uuid16>` |
| `workspace_id` / `owner_id` | `String(40)` **INDEX** | scope + ownership |
| `scope` | `String(20)` | `document` / `multi` / `workspace` |
| `document_id` | `String(40)` NULL **INDEX** | single-doc scope (Document.id) |
| `document_ids` | `JSON` NULL | multi-doc scope |
| `conversation_id` | `String(40)` NULL | summarize-a-chat (future) |
| `title` | `String(300)` | editable; auto-defaulted from type/subject |
| `summary_type` | `String(20)` | `quick\|standard\|detailed\|bullet\|chapterwise` |
| `language` | `String(20)` | multilingual future-proofing |
| `status` | `String(20)` **INDEX** | `queued\|processing\|completed\|failed\|cancelled` |
| `progress` | `Integer` | 0–100 |
| `stage` | `String(60)` | e.g. `section 2/5` |
| `error` | `Text` NULL | failure message |
| `model_name`, `prompt_version` | `String` | provenance |
| `token_usage`, `generation_ms` | `Integer` | telemetry (`generation_time`) |
| `section_count` | `Integer` | denormalized |
| `version`, `parent_summary_id` | `Integer` / `String` NULL | version-history future-proofing |
| `deleted_at` | `DateTime` NULL | soft delete |
| `created_at` / `updated_at` | `DateTime` | `updated_at` `onupdate` |

### Table: `summary_sections`
`id (sec_…)`, `summary_id` **INDEX**, `heading`, `order`, `content (Text/Markdown)`,
`citation_count`. Index `(summary_id, order)` for ordered reads.

### Table: `summary_citations`
`id (scit_…)`, `summary_section_id` **INDEX**, `document_id` (vector id → Module-3 `/by-vector`),
`chunk_id`, `page_number`, `workspace_id` **INDEX**, `citation_text`, `confidence`.

### Relationships, indexes & scalability
Logical FKs (`summary.workspace_id/owner_id`, `section.summary_id`,
`citation.summary_section_id`). Section + citation reads are **batched** (sections by summary, then
one `IN (...)` for all their citations → no N+1). Indexed for list (`owner_id,workspace_id`),
recency (`workspace_id,created_at`), status polling (`status`), and ordered section reads. Future
scalability: `version`/`parent_summary_id` (history), `language` (multilingual), a future
`document_scope` cache; Postgres by env var as elsewhere.

## 5. Backend Architecture

Clean-architecture package `app/summaries/` (logic never in handlers, SQL only in repositories,
**no faiss import**, **no separate retrieval**):

```
models.py      Summary / SummarySection / SummaryCitation ORM (3 new tables)
schemas.py     DTOs + list-query enums (sort/status)
validation.py  pure type/scope/title validation + default titles
errors.py      transport-agnostic domain errors (404/409/422)
repository.py  all SQL (owner+workspace scoped, batched citations, section clear)
engine.py      SummaryEngine protocol + PipelineSummaryEngine (section plan + per-section pipeline, lazy)
service.py     lifecycle + generate_now (persist sections/citations, progress, cancel, failure recovery)
runner.py      background execution (ThreadPool prod runner + inline test runner + deferred runner)
api.py         authenticated async routes under /workspaces/{id}/summaries
```

### Repositories
`SummaryRepository` — owner+workspace scoped, soft-delete aware; `list` in two queries with search
(title), type/status/document filters, sort, pagination; batched `sections` + `citations_for`;
`clear_sections` (for regenerate); `get_by_id_only` (the runner's own session looks up by id).

### Services
`SummaryService.create` validates type/scope/title and enqueues a `queued` row (bumps
`Workspace.summary_count`). `generate_now` — the pipeline — clears prior sections, flips to
`processing`, consumes the engine's events (`plan` → `section`* → `final`), persists each section +
its citations, tracks `progress`/`stage`, **checks for cancellation between sections**, and on
failure records the error while keeping partial sections (failure recovery). Also
`rename`/`cancel`/`reset_for_regenerate`(bumps `version`)/`duplicate`(copies sections+citations)/
`delete`.

### Background processing (queue / progress / cancel / retry)
`runner.py` provides `submit(summary_id)`:
- `SummaryRunner` (production) — a `ThreadPoolExecutor` worker opens its OWN `SessionLocal` session
  and runs `generate_now` with the real `PipelineSummaryEngine` (so the request returns instantly).
- `InlineRunner` (tests) — runs generation synchronously on submit with an injected session factory
  + fake engine (deterministic, no threads/faiss/LLM).
- `DeferredRunner` — no-op (leaves `queued`, for queue/cancel state tests).
Progress is polled via `GET .../status`; **cancel** sets status→`cancelled` (the worker stops
between sections); **retry** = `regenerate` (reset to `queued` + resubmit); **failure recovery** =
status→`failed` with the error, partial sections retained.

### Validation & error handling
Pure validation (`validation.py`): summary type ∈ the supported set, scope inference + requirements
(document needs `document_id`, multi needs `document_ids`), title limits. Domain errors
(`SummaryNotFound` 404, `SummaryStateError` 409, `SummaryValidationError` 422) mapped to HTTP by a
`_handle` translator.

## 7. AI Pipeline Integration

Every section of every summary flows through the **existing** pipeline — the module adds only
planning, orchestration, and persistence:

```
section query  →  build filter { workspace_id, doc scope (vector ids), exclude archived/deleted }
              →  RetrievalPipeline.run   (query analysis · dense + BM25 · RRF · rerank)   [Phase 1]
              →  ContextBuilderService.build  (dedup · evidence-rank · budget · compress · assemble) [Phase 2]
              →  build_summary_prompt(type, heading, engineered-context)  →  LLM (complete)
              →  structured_citations(evidence)  →  persisted SummaryCitations
```

The engine reuses `pipeline`, `context_builder`, `generate_embedding`, `build_filter`,
`structured_citations`, and Module-2's `list_excluded_vector_ids` (archived exclusion) +
`list_document_chunks` (heading derivation). The only additions to `answer_service` are
`build_summary_prompt` (per-type styling) and a small `complete(prompt)` raw-completion helper
shared with the QA path. **Duplicate detection, evidence ranking, compression, context assembly,
and citation preservation are inherited unchanged** — there is no second retrieval path, and whole
documents are never fed to the LLM (each section is a bounded, engineered context).

## 8. Hierarchical Summarization

Large documents (books, papers) can't fit a single prompt, so summaries are built **hierarchically**
— the "chunk groups → intermediate summaries → final summary" strategy:

```
Large document
   ↓  derive logical sections from the document's OWN headings (chunk metadata `section`)
Section plan  (capped at MAX_DERIVED_SECTIONS; falls back to generic themes if no headings)
   ↓  per section:  retrieve (scoped) → engineered context → summarize   ← intermediate summaries
Intermediate section summaries
   ↓  (detailed / chapterwise)
Final synthesis  =  summarize the intermediate section summaries into one "Overall Synthesis"
```

- **Per-section retrieval** means each intermediate summary sees only its slice of engineered
  context — bounded token cost, grounded output, and correct citations per section.
- **Final aggregation** runs one more pass over the intermediate summaries (a summary-of-summaries),
  yielding a coherent executive synthesis without ever loading the whole book into a prompt.
- **Type-aware plans:** `quick`/`bullet` = 1 section; `standard` = 3 fixed sections; `detailed`/
  `chapterwise` = derived headings (hierarchical) + synthesis.

**Performance benefit:** cost scales with the *number of sections* (bounded), not document length;
each LLM call gets a small, high-signal context; and sections stream into the UI as they complete.

## 9. API Documentation

All routes under `/workspaces/{workspace_id}/summaries`, require a bearer token, 404 if the
workspace isn't owned.

### Generate (async)
- `POST ""` → **202** `SummaryOut`. Body `{ summary_type, scope?, document_id?, document_ids?,
  title?, top_k? }`. Creates a `queued` summary + enqueues generation; bumps `summary_count`.
  Errors: 422 (unknown type; document scope without `document_id`; multi without `document_ids`).

### Status & details
- `GET "/{id}/status"` → `SummaryOut` (poll: `status`, `progress`, `stage`, `error`).
- `GET "/{id}"` → `SummaryDetail` = `SummaryOut` + `sections: [{ id, heading, order, content,
  citation_count, citations: [SummaryCitation] }]`.
- `GET ""` → `{ items, total, page, page_size, pages }`. Query: `page, page_size, search,
  summary_type, status(any|queued|processing|completed|failed|cancelled), document_id,
  sort_by(created_at|updated_at|title), order`.

### Actions
- `PATCH "/{id}"` `{title}` → `SummaryOut` (rename).
- `POST "/{id}/regenerate"` → `SummaryOut` (reset to queued, `version`+1, re-run).
- `POST "/{id}/cancel"` → `SummaryOut` (409 unless queued/processing).
- `POST "/{id}/duplicate"` → **201** `SummaryDetail` (copies sections + citations).
- `GET "/{id}/export?format=md"` → **text/markdown** attachment (title + sections + per-section
  Sources).
- `DELETE "/{id}?permanent="` → 204 (soft by default; permanent purges sections + citations).

### Validation & errors
`422` invalid type/scope/title · `404` unknown summary/workspace · `409` illegal cancel · `401`
unauthenticated.

## 10. Performance Optimizations

- **Background generation** — a thread-pool worker runs generation off the request path; the API
  returns 202 instantly and the client polls `/status`. Large/multi-section summaries never block.
- **Bounded per-section context** — each section retrieves a small, engineered context; cost scales
  with section count, not document length (hierarchical).
- **Reuse of retrieval** — one pipeline; no duplicated retrieval logic; archived docs excluded via a
  cheap indexed lookup.
- **Batched citation loads** — a summary's sections load all citations in one `IN (...)` query
  (no N+1).
- **Two-query listing** + indexed status polling; denormalized `section_count`/`progress` for O(1)
  card rendering.
- **Cancellation short-circuits** — the worker checks status between sections and stops early,
  freeing the pool.
- **Frontend** — polling with cleanup, debounced search, `AbortController`, memoized cards, lazy
  route chunk; markdown deps already loaded (no extra bundle cost).
- **Caching seam** — because retrieval is per-section and deterministic per (query, filter), a
  future context cache can memoize identical section retrievals across regenerations.

## 13. Future Compatibility

| Future capability | What Module 5 already provides |
|---|---|
| **Smart Notes** | a section (heading + grounded content + citations) maps directly to a note; the summary is a note source |
| **Flashcards** | sections + citations generate card front/back with provenance; "Convert to Flashcards" is a downstream action over persisted sections |
| **Knowledge Graph** | persisted `SummaryCitation` edges (summary → section → document → page → chunk) are graph edges; `by-vector` resolves nodes |
| **Research Reports** | multi-document summaries + the hierarchical synthesis are the building blocks of a longer, cited report |
| **Multimodal Summaries** | scope + engine are content-agnostic; a future image/audio extractor feeds the same section pipeline (sections stay text + citations) |
| **Agents** | the injected `SummaryEngine` interface is the extension point — an agentic, tool-using summarizer implements the same `generate()` event contract with no changes to persistence/runner/UI |
| **Version history / collaboration** | `version`/`parent_summary_id` reserved; `owner_id` real; a future memberships table shares summaries with no migration |

Every future capability plugs into the **`SummaryEngine` seam** (swap/extend the generator) or reads
the **persisted sections/citations** — the summary substrate itself doesn't change.

## 14. Lessons Learned

### Architecture decisions
- **Orchestrate the pipeline per section; never dump the document.** A summary is many bounded,
  grounded retrieve-and-summarize passes. This keeps token cost bounded, output grounded, and
  citations correct — and reuses the one retrieval path in the system.
- **Async via an injected runner.** Generation runs on a thread-pool worker with its own DB session;
  the API returns 202 and the client polls. The runner is injected, so tests run it inline and the
  whole pipeline is deterministic without threads/faiss/LLM.
- **Hierarchical by derived headings.** For deep types, sections come from the document's *own*
  headings (intermediate summaries), then a synthesis pass — real hierarchical summarization that
  scales to books.
- **Event-driven service.** The engine yields `plan → section* → final`; the service persists and
  tracks progress from those events, so streaming/async and the fake engine share one contract.
- **New tables, additive shared changes.** Three new tables + two new `answer_service` helpers; no
  existing endpoint changed shape.

### Tradeoffs
- **Thread-pool background** (not Celery/Redis) — keeps the offline-first, zero-ops stance; swap in a
  real queue behind the same `runner.submit` contract when horizontal scale is needed.
- **Polling** (not push) for progress — simple and proxy-friendly; an SSE progress stream can back
  the same status model later.
- **Heuristic token accounting** for `token_usage` (as in chat) until the Ollama HTTP API gives exact
  counts.
- **Snippet-based citations** per section — robust and clickable; exact glyph ranges are a future
  refinement shared with the viewer.

### Known limitations
- Section plans for `standard`/theme fallback are fixed queries; a smarter planner (LLM-proposed
  section outline) would improve coverage — a drop-in engine change.
- No cross-regeneration context cache yet (each regenerate re-retrieves); the per-section determinism
  makes that a straightforward future optimization.
- Multi-doc synthesis merges via retrieval over the selected docs; a dedicated cross-document
  dedup/merge pass could tighten long briefs.

### Future improvements
1. LLM-proposed section outlines (smarter plans) behind the same engine seam.
2. Context/result caching keyed by (query, filter) to make regenerate near-free.
3. A real job queue + SSE progress stream for very large batches.
4. "Convert to Notes / Flashcards" actions over persisted sections (Modules 6+).
5. Exact-range citation highlighting and export to PDF/DOCX.
```
