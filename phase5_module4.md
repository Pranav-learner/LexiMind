# Phase 5 ¬∑ Module 4 ‚Äî Audio & Video AI Workspace

> The **product-integration capstone** of Phase 5. Every prior module built temporal *infrastructure*
> (processing ‚Üí intelligence ‚Üí retrieval ‚Üí context). This module fuses them into one seamless
> experience: upload a lecture, ask a question, get a **timestamp-aware answer with citations that
> seek the player** ‚Äî while all intelligence keeps flowing through the existing Retrieval Engine,
> Context Engine, and the single Answer Service. Like the Phase-4 `mmworkspace`, it is a **pure
> orchestrator**: it coordinates existing services and duplicates none of them.

---

## 1. Module Overview

**Why temporal AI needs a dedicated workspace.** A recording is only useful if a human can *interact*
with it: watch it, jump to a moment, read the transcript, ask a question, and turn it into notes.
Modules 1‚Äì3 made a lecture *understandable* by machines; this module makes it *usable* by people. It
is the surface where documents, images, and recordings coexist and where the user never thinks about
ASR, diarization, timeline retrieval, context engineering, or prompt building.

**How users interact with multimedia knowledge.** One screen: a media library, a synchronized player,
an interactive multi-lane timeline, a live transcript, a temporal-grounded AI chat, and an AI action
panel. Ask *"what did the professor say about deadlocks?"* ‚Üí get an answer whose citations are
timestamps you click to seek. Press *"Study Guide"* ‚Üí a grounded note is generated by the existing
notes service.

**Product vision.** LexiMind handles a lecture as naturally as a PDF ‚Äî same workspace, same chat, same
knowledge assets (summaries/notes/flashcards), same citation model ‚Äî now extended along the time axis.

## 2. Previous Architecture

Before this module the temporal backend was complete but **only reachable as separate, developer-facing
surfaces**: the media processing dashboard (M1), the temporal-intelligence endpoints (M2/tintel), and
the temporal search + inspectable prompt (M3/tretrieval). There was:
- **no unified media workspace** ‚Äî recordings and documents lived on different pages;
- **no media chat** ‚Äî temporal retrieval produced a *prompt preview*, never an actual answer (M3 was
  deliberately LLM-free);
- **no synchronized player / timeline / transcript** ‚Äî no way to *navigate* a recording;
- **no timestamp-citation navigation**, no media-driven knowledge-asset generation, no media telemetry.

The pieces existed; nothing connected them into a product.

## 5. Backend Architecture

**`app/mediaworkspace/` ‚Äî pure orchestrator (1 observability table, no business logic):**
- **`engine.TemporalChatEngine`** ‚Äî implements the existing chat-engine interface
  `generate(content, workspace_id, history, *, db, top_k, document_scope)`. Derives owner from the
  workspace row (so the chat interface is reused *unchanged*), runs `TemporalRetrievalService.search`,
  calls the injected `answer_fn` (prod = `answer_service.complete`; tests = fake), maps temporal
  citations into the chat citation shape, and stashes the rich citations on `last_result`.
- **`service.MediaWorkspaceOrchestrator`** ‚Äî `overview`, `library`, `unified_timeline` (ensures tintel
  derivation, merges lanes), `playback_meta`, `media_chat` (drives `ChatService.run_message`),
  `ai_action` (routes to summaries/notes/flashcards via `_ACTION_MAP`), `search` (temporal ‚äï
  multimodal), `record_interaction` / `observability`.
- **`models.MediaInteractionEvent`** ‚Äî the single telemetry sink (Step 15), consistent with every
  module owning one log table.
- **`repository`** ‚Äî only the interaction table; everything else reads through existing domain repos.
- **`api`** ‚Äî `/workspaces/{id}/media-ai/*`; the chat engine + asset runners are injected via `Depends`.

**Routing & pipeline selection.** The orchestrator never re-implements retrieval: `media_chat` always
selects the *temporal* pipeline (`TemporalChatEngine`); `search` fans out to both temporal and
multimodal engines and merges. **Caching/validation/errors** reuse the underlying services
(`ensure_derived` guards derivation; domain errors carry `status_code`; `_ACTION_MAP` validates
actions ‚Üí 422). The document file/media stream reuses the existing authed endpoint.

## 10. Performance Optimizations

- **Lazy media loading** ‚Äî the player fetches the file as a blob **only** for the selected recording;
  object URLs are revoked on switch.
- **Single position source of truth** ‚Äî one `positionMs` state drives timeline highlight, transcript
  highlight, and auto-scroll (no redundant re-renders/queries).
- **Parallel loads** ‚Äî playback meta + timeline + transcript fetch concurrently on selection.
- **Derive-once** ‚Äî `unified_timeline` uses tintel's count-guarded `ensure_derived`; searches reuse the
  same guard, so temporal intelligence is built once per recording.
- **Bounded reads** ‚Äî library/timeline/search all cap their candidate reads; the timeline is CSS-only
  (no canvas) so long lectures render cheaply.
- **Fire-and-forget telemetry** ‚Äî interaction posts never block the UI.
- **Streaming-ready** ‚Äî the chat reuses the chat pipeline, which already exposes an SSE path for future
  token streaming of media answers.

  streaming transcripts and live timelines.

## 14. Lessons Learned

**Architecture decisions**
- *Reuse the chat pipeline by injecting an engine, not by forking it.* Because `ChatService.run_message`
  already delegates the answer to an injected engine, media chat needed **zero** changes to chat ‚Äî
  a `TemporalChatEngine` that derives owner from the workspace slotted in behind the exact interface.
  This finally wired the LLM that Module 3 intentionally left as a prompt preview.
- *Mirror `mmworkspace`.* The capstone pattern (pure orchestrator, coordinate don't compute, route AI
  actions to existing services + runners) transferred directly to the temporal domain.
- *One observability table, not zero.* A single `MediaInteractionEvent` sink captures the front-of-house
  interactions (playback/seek/citation-click) that have no existing home ‚Äî consistent with every module
  owning one log table, without duplicating business logic.
- *One `positionMs` on the frontend.* Making playback position the single source of truth kept the
  player, timeline, and transcript trivially in sync.

**Tradeoffs**
- Knowledge-asset generation (summary/notes/flashcards) reuses the existing generators, which retrieve
  via the FAISS text pipeline ‚Äî so for a *media* document their grounding is currently limited (the
  media chat is the deeply temporal-grounded surface). Wiring those generators to temporal context is a
  clean future enhancement once transcript embeddings land (the `embedding_status="pending"` queue).
- Media chat is non-streaming here (the SSE path exists in the reused chat pipeline and can be enabled).
- Timestamp citations ride inside the reused `MessageCitation` (start_ms encoded in `chunk_id`) to avoid
  a second chat store; the orchestrator returns the rich temporal citations alongside for the UI.

**Known limitations & future improvements**
- No live streaming yet (player is file-based); the seams (SSE, incremental derivation) are in place.
- Deep temporal grounding of generated assets + token-streaming media answers are the next steps, both
  additive. Phase 6 agents build directly on this orchestrator.
